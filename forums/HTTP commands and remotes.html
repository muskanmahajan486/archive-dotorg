After playing a bit with the on-line designer and looking at some demos, I've learned that remotes can communicate with IP devices using proprietary ASCII commands through Telnet, TCP/IP and HTTP. This is great since many IP controllers provide a way to be monitored and controlled via a REST interface. I too appreciate the availability of an on-line tool for designing GUI layouts. Honestly guys, you've don an excellent job. Congratulations!

I have some questions regarding the remote interface:

1. If I create a sensor and want to display the value returned by this sensor, is there any way to configure a periodic update on the remote? TCP and HTTP commands would need to be resent periodically in order to get the latest value update from the sensor I guess. I'm primarily thinking in a default screen, continuously showing temperatures from different locations.

2. Are HTTP commands directly sent from the remote or do they pass through the controller too? Same question for TCP connections.

Thanks again!

Daniel.

Comments:

1) Depending on the protocol the command for the status update is executed every x seconds. If the status changes the controller updates it's cache and new value is displayed on the remote. This is already available.

2) Everything is passing the controller.

Posted by dberenguer at Sep 29, 2011 18:28

The HTTP protocol can be used for sensors but it only returns the repsonse from the HTTP request. If your service/device answers with plain text eg "ok" you can use this. You can also create a custom sensor and have the result in there but no parsing is supported in HTTP protocol yet.
