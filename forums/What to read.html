Thanks,

We are working on documentation and video tutorials to help people get started (what equipment is needed, how to create designs and so on). At the moment most documentation is reference material.

'Test UI' is still a bit untested (pun) in the designer, will work on making it better.

TCP/IP is just for TCP at the moment. UDP would not be a difficult addition for one-way commands if you're looking to contribute (Java provides pretty well documented UDP APIs).

The one thing about both UDP/TCP/HTTP generic connectors is that they're mainly suitable to one-way commands at the moment. For feedback, you are likely to need a device specific protocol implementation (e.g. Yamaha HTTP response parsing and so on) which at the moment is a Java implementation. We're working on getting scripting bolted in here to avoid low level programming access requirement.

Early days for us too but are happy to receive any help whether testing/user feedback/protocol implementations/documentation.

Juha you did start to elaborate around the area I was trying to understand. How does the panel get feed back? This is one of the many areas that differentiates OR from other solutions, lets say a normal remote. A normal remote can't tell what the current volume of the TV is, only that you can turn the volume up or down. In theory. There's a lot of ifs and buts with this though. One is that most infrared control is one-way only, so you can't get an exact status of what the device is in (such as TV volume). It looks like this year IP connections to top-range TVs are becoming pretty standard, so there's hope for better control (and then there's Google TV). There are some TV brands that allow serial connections also which gives bidirectional control.

So at the end of the day, we too have to live with whatever control interface the hardware manufacturers offer. And that there's no standardization there means a simple issue such as volume control can become a pain in the neck (unless you go for very specific models where you know you have controls beyond basic infrared).

For my simple scenario for example, I'd love to glance down at the panel and see that the MythTV box was off - press a button, one WOL later... it's on. Or my partner might have already switched it on using the traditional remote, but the smart panel knows this and reflects that the device is on through the user interface. This is such a leap forward from today's remotes.

MythTV does have an IP interface if I recall correctly, I think Marcus Redeker used it with his Telnet protocol plugin already.

Since MythTV is such a popular product, I would be interested in adding a dedicated protocol support for it (rather than everyone having to set up generic IP protocol commands) and adding read commands for status where available. So if you fancy some protocol coding in Java, let us know.

Same applies to the Yamaha protocol Pierce is talking about. These are fairly simple to implement, just need to take the time to type them down (and currently requires some familiarity with Java, will be adding scripting here later).
