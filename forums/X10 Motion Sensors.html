Posted by mredeker at Jan 07, 2013 10:34

The warning on X10 looks like a false positive, caused by a separate change that automatically adds a 'name' property to all commands. The X10 implementation hasn't been updated to reflect this change so it warns about an unrecognized property ('name') but should carry on as normal.

The X10 implementation should be updated to remove this false warning.

The debug statements about <state> elements are nothing to concern about (and that's why they are recorded at debugging level) – the controller notices an inconsistency in the controller model caused by our current tooling (this happens regardless if you configure things right or wrong) and the debug statement is there just to indicate that the controller noticed this and remedied the issue.

So based on these logs, you're good to go (there's always a possibility of other errors of course not shown here).

Posted by juha at May 27, 2013 10:26

Based on this discussion last January, I decided to develop my own X10 interface using Python, so it could be easily moved to a RPi. My Python-based system now resides on a Windows 7 computer; I will eventually move it to a RPi, but haven't yet because the RPi I bought for this is now used 24/7 for Pandora.
During this project, I've made several observations, the most notable being that X10 is patently unreliable. Anecdotal evidence suggests that this is a result of random noise on the power line being misinterpreted by the CM11A; for example, after running flawlessly for many days, the system locked up following the use of a paper shredder, a device known for generating power line noise. When X10 was designed in the 60's, power lines were relatively noise-free; such is not the case today. The Internet is loaded with discussions about why CM11A's lock up, and the various incantations that can be used to remedy or prevent these lock ups, but the bottom line is--random noise on the power line can cause the CM11A to lock up, and once it is locked up, there is no way for software to unlock it (until now; more below).
Many of the lockups can be corrected by valid communications on the power line, a condition that occurs naturally in a system that uses motion detectors. I can detect when the CM11A is locked up, because my system sends a phony command every 5 minutes, and when the CM11A doesn't respond properly, the system sends me an email message reporting the communications failure. Once it has reported this, if/when the CM11A once again becomes responsive, it sends me another email message reporting the resumption of normal operation. Again anecdotally, I've determined that the correction is usually a result of my cat walking near an extra motion detector whose sole function is to create random valid X10 commands on the power line.
A system that includes a cat with insomnia as a key component can hardly be described as reliable, so I sought a more reliable solution, and discovered this: http://www.numato.com/1-channel-usb-powered-relay-module. Since my system can detect when the CM11A becomes non-responsive, and cycling its power always resets it, I now have a method for the software to unlock the CM11A.
Since I've written the software to satisfy my needs (turn lights on and off depending on time, dawn & dusk, be notified when my cat wants to come inside, email myself a webcam shot when someone drives down my lane, send my wife a text message when the greenhouse windows need to be opened), it would never fit into the 'general purpose' category. And as I mentioned with regard to my Java programming expertise, I can write Python programs, but I am not an accomplished Python programmer.
My advice for OR; don't waste a lot of time implementing the code to communicate with the CM11A. Yes, it's pretty straightforward under ideal conditions to turn lights on and off, and to determine if a motion sensor has detected motion, but these days conditions (as related to the integrity of the power line) are rarely ideal. I'm now convinced that an implementation that doesn't accommodate power line noise is a system that will inevitably periodically lock up, and I doubt that many will be willing to go to the lengths that I have (cycle the power to the CM11A) to ensure reasonable up-time.
All the best,
Tom
