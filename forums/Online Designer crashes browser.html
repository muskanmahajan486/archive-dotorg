Posted by juha at Jun 12, 2011 16:29

There's an alternative version deplyed at http://designer.openremote.org/2.0-SNAPSHOT-20110612/

This is just a stop-gap that improves the performance somewhat but does not eliminate the issue. A proper fix will require further work in analyzing the SQL queries and the data transmission between the DB and designer and maybe even investigating the Javascript generated by the GWT and GXT frameworks.

There seem to be a couple of issues at work here. One is the database query performance – some minor improvements can be had by not eager-loading data that is not needed and doing lazy fetch strategies instead. However, the schema structure around sensors looks a little odd so refactoring this area may help (selecting commands does not suffer similar performance issues and the designer code there takes a different approach from sensor loading).

But also updating the GXT UI framework had an impact on performance so it may not merely be a DB performance issue but Javascript performance issue as well (there shouldn't be any because the displayed UI is very simple but for some reason the Javascript ends up doing a lot of work).

So the performance will be somewhat dependent on the browser you use – this seems to be working ok with the Chrome on Macbook Pro. Firefox on the same Mac complains about a slow script – stopping the script and trying to select a sensor again eventually resolves it (the dialog appears) but is quite a bit slower compared to Chrome so you'll have to wait quite some time.

Also the GXT update broke at least some of the other UI code which will need to be updated before these changes can be deployed on the main version. The proper fix will have to wait for later until have time to do all that is described above.
