From what I've read KNX defines a standard for interfacing with and controlling a device. It also provides different options for physical connectivity between these devices, however it's more about the interface and control protocol - is this correct? CBUS is similar in that it specifies a physical connectivity medium (over UTP or proprietary wireless) and defines a interface and control control standard between all CBUS capable devices. Clipsal, the manufacturer of CBUS here in Australia, announced in April 2009 that they are publishing the serial protocols for free use by developers. Have a look at http://www.cbus-enabled.com/news-8.htm for the full details. I'm positive that having these protocol documents available will make interfacing OR with CBUS much more palatable than before. Is there anything, besides the widespread EU adoption of KNX, that KNX has over CBUS?

It's probably noteworthy to mention that my plan was to always interface with CBUS via their C-GATE tool which bypasses the requirement to talk native CBUS. C-Gate is essentially a freely available daemon written and maintained by Clipsal that lives on a box that maintains a constant connection to the CBUS network either by IP or RS232. It can be used to instruct the CBUS network to perform various actions, e.g. turn on lighting group X, using formatted commands and responses via a telnet interface. C-Gate also has a number of options to allow third party applications to receive a continuous real time list of events that occur on the CBUS network. All this info came from: http://www.clipsal.com.au/trade/__data/page/81/C133.pdf.

So, while the serial protocol specs are a great step forward in having totally accessible standards for CBUS that give you the option to create your own interface, in most cases you could probably just use C-Gate to do everything. There are a few exceptions, such as adversity to having a separate interface daemon that you have to run and rely on. Most of the flack I've read about C-Gate I've read focused around it's reliability, or lack thereof. It's seems to have matured in versions since those discussions though.

The concept design I have for my automation system would utilise a GC100 serial or IP controller as opposed to a commandIR system. Are you able to tell the commandIR system to do something with one of it's IR emitters over an IP or serial connection, rather than having to do everything via remote control?

We are approaching this slightly differently. The ORB is the controller and CommandIR (or GC for that matter) is an extension to it. The logic (our code) is in the ORB which then uses the extensions over various different media (IP, RS-232, USB). For the CommandIR example, we use its USB driver to control it from the ORB. CommandIR itself is not an IP-enabled device.

The ORB is an IP device, part of your home LAN. Our current reference implementations also include WiFi and the ORB responds to HTTP requests over IP network.

There are some cases (IP-based gateways) where we can control things directly from IP-enabled consoles (iPhone etc). For example, GC or IRTrans can be controlled directly via IP, many other products too. For some scenarios ORB is not necessary although we suspect over time having logic deployed on a controller (such as ORB) to react to events coming from other devices will change this. A lot of the higher end functionality that Neil and Dan C discuss are examples of this.
