Screen orientation and navigating with the tab bar seem to have a compatibility problem when running on an iPod Touch or iPad using the iPhone app. I don't have an iPhone, so I don't know if the behavior is the same with the iPhone. Here is the issue:

I created several screen groups and setup the tab bar with some navigation items -- different tab bars for each group. When setting the navigation link for the tab bar items it allows me to choose to navigate to either the portrait or the landscape version of screen-- but not both and not automatically to the one that corresponds to the current orientation of the device. So, I linked them to the portrait version by default. The destination screens have the orientation option set to be both portrait and landscape and I've used the portrait and landscape tabs at the bottom of the designer page to create both versions of the screens.

Now, if I am running in landscape mode and I touch a navigation button in the tab bar, it takes me to the portrait version of the destination screen. Furthermore, the the screen remains in portrait mode even though the device is being held in landscape mode. To get it to switch to landscape mode, I have to rotate the device to the portrait orientation and then back to the landscape orientation.

This makes the portrait and landscape setting for a page essentially useless if you are using tab bar navigation.

Ideally, tab bar navigation should be smart enough to go to the proper orientation of the page without me having to specify it in the navigation link, especially since there is no way to specify different orientations in a tab bar in a group based on the current device orientation.

Does anyone have any recommended work arounds to this issue?

This raises an interesting "issue".
Currently, this bar is called a Tab Bar and on the iPhone/iPad, it maps to an iOS Tab Bar.
Looking at the Human Interface Guidelines (HIG) from Apple, one can see that:

  • tab bars always appear at the bottom of the screen
  • tab bars should be used to switch between views / modes and not to perform actions
  • other indications such as limit on number of items, consistency amongst screens, ...

Looking in the OpenRemote designer, it is currently possible to add a navigation such as next screen as a TabBarItem, which goes against the "no action" rule of the HIG.

So in the longer run, should we foresee an OR specific bar?
This can be more powerfull with features such as:

  • can be placed on any side of the screen
  • can be made to show/hide with a tap on the screen (like an iOS toolbar e.g. in Photo app on iPhone)
  • user could swipe to reveal more icons
  • could be made into a sort of PlayStation XMB (XrossMediaBar)

1) We may need a 'cross-platform' tab bar eventually – the iOS has limitations (for example the rules on images) that do not make sense elsewhere, on Android for example. I think the latest Android templates (Honeycomb) also allow placement on different sides of tablets (not sure what iOS offers there) and finally that on some systems tab bars are always visible, on others they maybe hidden from view when not used.

So having consistency there would help.

2) We will need with certainty additional properties on both designer and panels about which system the panel is being designed for – if it is an iOS panel and the native tab bars are desired then impose the necessary limitations on the designer on what can / cannot be done with the tab bar (or any other component) or limit the set of components depending what system the panel is being designer for (iPhone vs. iPad for example).

So these properties (metadata) can be used in the designer to control what can be done, and the user choices limited to what makes sense for that particular type. It can be used also on panels to pick appropriate screens from controller at startup to include only the "correct" types and panel options.

So this is something that should go into the discussion of the next panel schema, an extendable <property> element that can direct panel behavior.
Posted by juha at Apr 19, 2011 11:45

I did implement a fix for this. When navigating to a specific screen, console checks if a "more appropriate version of that screen" exists for the current device orientation and uses that instead if it does.
This means that if you have both a portrait and a landscape version of a screen and in a navigation (tab bar, button, gesture, ...) you link to the portrait screen but your device is in landscape, the console will jump to the landscape version of the screen.

The fix can be found in this branch : https://openremote.svn.sourceforge.net/svnroot/openremote/workspace/ebariaux/iPad_Console_2_0_0_Betas_Linphone

Note that this branch is used for implementation of other fixes / cleaning and might not always be stable. However, if you're a brave soul, please go ahead and provide any feedback you have.
