Folks,

I've spent some time trying to improve some of the issues with the existing Android Console and now 2.1.0 beta is available to try: -

Android Console 2.1.0 Beta

Some of the notable changes: -

  • Various improvements to general code and stability
  • System Screens updated (now look a lot like the Webconsole system screens)
  • Controller availability indicator added
  • History management improved
  • Slider re-written to avoid display issues with varying display densities
  • Minimum Android version increased to 7 (Android 2.1) - necessary for some of the functionality now used

Usage: -

  • Controller auto-discovery is automatic
  • Load a controller by tapping it in the controller list
  • Remove controller's by holding down on them in the controller list and selecting delete when prompted
  • You can edit a controller by clicking the '>' image on the right hand side (Username and password is still to be implemented but default panel works)
  • Only controller's that are available can be loaded

A branch with the code can be found at: -

https://openremote.svn.sourceforge.net/svnroot/openremote/workspace/richturner/tag/Android_Console_2_1_0_Beta

Please leave feedback here.

Rich

OK, I have found what I think is a reasonable fix to my issue and thought others may appreciate it.

I got the source for "Android_Console_2_0_0_Beta3" and made a change to the "PollingHelper" class. It turns out there was a SocketException caused by the brief disconnection caused when wifi is switched to 3g/4g and vice versa. This appears to be a problem with how network exceptions are handled. When a SocketException occurs the code doesn't seem to cater for a split second outage. It looks like the issue is with the ORConnection.Java or ORNetworkCheck.Java. What these classes seem to do is try all possible reasons except there not being a connection. They eventually result in a thrown error which crashes the app.

I didn't have time to go through this code to fix this logic, so I have made a change to the PollingHelper.Java file which is the only place that I have seen affected by this issue.

The code I edited is in the "handleRequest" method. I have commented out two lines of the SocketException catch, shown below....

}catch (SocketException e) {
if (isPolling)

Unknown macro: { //isPolling = false; Log.e(LOG_CATEGORY, "polling [" + pollingStatusIds +"] failed.", e); //handler.sendEmptyMessage(NETWORK_ERROR); }

}

This prevents the app from crashing and allows polling to continue when SocketExceptions occur.

If anyone has a better way of fixing this, then great. Please let me know.
