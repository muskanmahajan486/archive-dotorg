Hello Greg,

Thanks for your offer to help. Java + C sounds like an ideal combination.

There are couple of ideas I think could be useful to start prototyping with.

Web Console

One is the web console. While our initial 1.0 release targets iPhone as the panel ("remote"), there's a need for a generic web based (HTML+CSS+jQuery, GWT if necessary) console to be used with other devices, smart phones, touch screen netbooks, regular old PC's as the panels for OpenRemote system.

We are right now working on getting the very first controller on the ORB (we should call it "Skinny" cause the only thing it does is translate REST to LIRC irsend). It takes the simplest possible REST command which you can see in the controller API documentation. So basically just a POST from a button.

The button id is coming from the configuration file bundle that you can download from http://composer.openremote.org/demo. If you peek inside this zip you'll see there's a controller.xml there which basically tells you what button IDs it expects and have mappings for. The button ID then maps to an event ID that can be infrared, x10 or knx which in turns maps to a specific command to those protocols.

In addition, you can look into the iphone.xml, it has some additional layout description on where the buttons go on iPhone layout. Whether this matters or not for the web console has yet to be seen.

So that's one idea. Basically a run-of-the-mill web app that hooks into the specific configuration files and REST API we are exposing in the OpenRemote system.

I suspect it needs a lot of details to be hashed out which is why I am suggesting it as a prototype – sort of to ensure the APIs we are developing take into account the specific requirements that a non-iPhone consoles might have and work through and evolve things as necessary to make things generic enough to support all additional panel development.

The Bare Metal Option

If you fancy more the Work Of Real Men (just kidding), then I've been glancing through the libusb APIs a few weeks back for the inevitable need we will have to talk to USB devices from Java layers. Couple of options here, go through existing work on Java-USB API and see if there's anything there that has a valid Open Source license, deploys on Linux (Voyage and Ubuntu at least) and is of acceptable quality. Or just whip together a simple socket based C daemon with custom IP protocol and talk libusb API directly. There's plenty of lib and framework scaffolding for all the socket part, no need to reimplement the wheel, and libusb is pretty straight forward too, just needs the integration protocol to be defined (idea similar to AJP httpd - tomcat bridge). Should be pretty quick job either way.

Again, don't have a specific USB device yet that needs is Right Now (but X10 CM15A interface comes to mind very quickly) but would be cool to have somebody look into it in preparation (especially the CM15A part).

So just couple of quick ideas, I should move them to wiki and put some more meat around it but right now a bit busy getting code released. But maybe it can get something going. Would really love to see somebody to take a bit of a lead and organize efforts around these.
