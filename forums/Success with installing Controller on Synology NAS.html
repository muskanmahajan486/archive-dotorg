you have to make a startup script and put it in usr/local/etc/rc.d

name it for example startOpenRemote.sh

#!/bin/sh

  1. run openremote at boot

export JAVA_HOME=/usr/local/java

cd /usr/local/OpenRemote/bin

case "$1" in
'start')
/usr/local/OpenRemote/bin/openremote.sh start
;;
'stop')
;;
*)
echo "Usage: $0

Unknown macro: { start | stop }
"
;;
esac
exit 0

The text changes when I save my message on the forum.
This file will start openremote everytime you reboot the synology.

It took me some time to figure out how you do it right. I had some problems with permissions and that the file wich were synced were not in the correct directory.

We'd be happy to support larger professional KNX installs – we will need you guys to tell us your requirements.

I assume IProuter1 here is a DNS lookup name or IP address of a specific gateway?

We can build additional semantics to the KNX group address with relatively low cost as a quick solution. In generic terms, multiple IP gateways will fit nicely with Eric's ideas on a "connectivity layer" in the designer (where IP gateway at address X.X.X.X:NNNN is a connectivity device and each KNX command is then associated with that particular gateway rather than universally) : Making connectivity layer explicit and simplifying device configuration

When auto-discovery is used, the mechanism probably needs to be tweaked a little (if it works as earlier, it only picks the first KNX IP gateway it finds).

Hi,

In addition to changing the webapp.port in config.properties, you will also need to change the port of the HTTP connector in Controller/config/server.xml file.

Ok guys,

I've managed to get the Open Remote controller working on my Synology DS211J!

Time for me to contribute something to this forum. I'll explain how I've done it. I didn't find it very easy since my knowledge of Linux is very limited (and the Synology DSM basically runs on Linux).

Let's start with a disclaimer: you are modifying your Synology beyond what is allowed by Synology. This might impact your software-warranty.

Step 1: Install the bootstrap script on your Synology
-----------------------------------------------------
This process is described on this page. Please follow the instructions with care and download the correct bootstrap script. Synologys use different CPU's, each CPU requires a different bootstrap.

Step 2: Install the Java Engine on your Synology
------------------------------------------------
Open Remote Controller requires Java version 6. The instructions and Synology package on this page will help you getting Java installed. As explained, before executing the Synology package, you need to download the java software from Oracle (this page) and the "DSM Toolchain". The guide refers to the toolchain of DSM 3.1, but the Toolchain for DSM 3.2 is also available (see this page). The JAVA_HOME variable should have been set automatically by the Synology package you have installed.

Step 3: Install the Open Remote Controller on your Synology
-----------------------------------------------------------
Copy the Open Remote Controller software (can be downloaded here) to the directory on you Synology where you want to run it from. I created a new folder "software" for this (which is technically "/volume1/software"). I'm not sure if there is a best practice for this. As I said, I'm a Linux-n00b.

After that, you could just navigate to the right folder and try to run Open Remote Controller, as a first test. This can be done from the Command Line Interface (I'm a Windows user, so I use 'Putty'). For me, this is:
cd /volume1/software/openremote/bin
sh openremote.sh run

After executing, I saw that port 8080 is already taken by another Synology service. So you would need to change that to another one (I chose '8081'). You could ofcourse also change the port of the other service you were running. You change the port like this:

  • In controller.xml, change the value of "webapp.port" to your new port (e.g. 8081). You can also find this parameter in the online designer and change it from there. I guess that's even better, because this value will be overwritten as soon as you sync with the online designer.
  • In server.xml find "port="8080" and change it to your new port.

Restart the open controller application, it should work now.
I've also encountered a "panel.xml not found" error if the pwd (present working directory) was not the same directory as where I've installed the Open Remote Controller. E.g. you are in /opt/etc and from there you launch the Open Remote Controller using "sh /volume1/software/openremote/bin/openremote.sh start". Panel.xml is really there, but the openremote controller is just looking in the wrong place.
I solved it by adding the line "cd /volume1/software/openremote/bin/" to the top of the openremote.sh script you can find in the "bin" folder. Ofcourse you should change that to your directory folder.
After solving these issues, I was able to execute the Open Remote Controller on my Synology.

Step 4: Automatically start the Open Remote Controller when booting DSM
-----------------------------------------------------------------------
We now have a working Open Remote Controller application that we can start from the CLI. However, when closing the CLI-session, the Open Remote Controller also shuts down. I wanted to run it in the background when my NAS is running, as a real Homeserver.

I got inspired by what is described on this page. I know it's a long page, but it gets interesting where it reads "Now we'll create the Serviio daemon start and stop script:".

Obviously, we will use a user "openremote" instead of "serviio" and also change the directories the where you have installed the open remote controller. Also put the commands "start" and "stop" on the right place.

Before executing the script, you should also create a user called "openremote" (this can be done in the DSM). Make sure it has read/write/execute access to the openremote folder. You also need to change the openremote user settings in the file "/etc/passwd", this is also explained on the same page (where it reads "In the DSM User Control Panel create a new user called serviio").

After executing all these steps, reboot your NAS and see if your (Android) device can locate the controller on your NAS.

Good luck!
