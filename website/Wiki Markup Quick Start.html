Headers

Use of headers:

h1. Title Level 1
h2. Title Level 2
h3. Title Level 3
h4. Title Level 4

Will render as:

Title Level 1

Title Level 2

Title Level 3

Title Level 4

  • You shouldn't normally need more than 4 levels of headers
  • Use Level 1 for page title
  • Use level 2-4 for sub headers

Page Separator

You can get a page separator with markup:

-----
  • You should avoid using page separators in most cases
  • If you find you need page separators, it may be time to break your page to multiple pages (pages can have child pages)
  • Shorter pages are better (easier for users to read)

Linking

Internal links on the website should use wiki linking format - not direct URLs. The former are updated automatically when pages are renamed or moved.

Link markup:

[text|space:pageName^attachment]

A simplest form of link is

[PageName]

While short, this simple format has couple of issues:

  • if pages are moved between spaces, it won't get automatically updated (it implicitly links to page 'PageName' in the same space which may not be available in a new space leading to a broken link
  • the link text is the same as page name, so 'PageName' which is often not ideal when linking something in the middle of the sentence

First issue can be fixed with explicit space name declaration (but leads to longer link markup meaning more boring typing):

[docs:PageName]
  • Above is resilient if page is moved to another space (the page in new location will still explicitly link to 'docs' space and its 'PageName' page). If 'docs' is not specified, moved page will attempt to link to 'PageName' in its containing (new) space which can be a broken link or a different page altogether (space is a namespace of pages)
  • Currently most documentation pages are in 'docs' namespace on this Wiki
  • Finding existing page links can be done from the little link button on the task bar of the page edit window. It lets you search for pages and will show the full page space and name when you find the page you were looking for. Very useful.

Second issue of embedding links into text is solved by following markup:

[You should go visit this page first.|docs:PageName]

This creates a sentence which links to the given page.

Linking to Attachments

Linking to attachments is done with:

[Click here to download.|docs:PageName^Files.zip]

Attachments are useful for larger images that don't fit into page layout, or videos and other media files, source code files, configuration files, XML files, etc. Short markup will also work:

[^Files.zip]
  • But can have similar maintenance issue (although attachments move with pages) as short version of link markup so usually the longer format is preferred (although requiring more finger work – but you'll save the work for the next person who needs to maintain the wiki on your behalf).

Linking Images

Creating image links can be done with following:

[!ImageName.jpg!|docs:PageName]
  • The above is a combination of the image markup and link markup. Images must be attached to a page first.

Reusing the same image across multiple pages can be done with following:

[!docs:PageName^ImageName.jpg!|docs:AnotherPage]
  • This long format fetches image from any other space and page that has the image attached and links the image to 'AnotherPage'.

Links to Page Anchors

Sometimes it is useful to link directly to a subsection of a page (especially if the page is long). This can be done for pages that use Headers efficiently and correctly.

The markup in the simplest form to a page subsection in the same page is as follows:

[#Headers]

Which will link to a page subsection that has h2, h3, or h4 header with title 'Headers', and renders as follows:

Headers

Adding a link text can be done as usual:

[This is where headers are explained|#Headers]

Will render as:

This is where headers are explained

Linking to another page's sub-section is also following the already established notation:

[Read More About Linking Images|website:Wiki Markup Quick Start#Linking Images]

Will render as:

Read More About Linking Images

  • As usual, for sake of robustness as the wiki and pages evolve, this last long markup format is preferred even if it means more strain on your fingers.

Call-Outs

Use {info} for informational messages, {note} for important messages.

They'll render as follows:

This is a callout for a topic that might be slightly out of the context from the main page content
This is a callout that the user definitely should be attention to!
  • Use both markups sparingly as they are visually very strong and using too many will reduce the readability of your page

Images (Layout)

Image alignment can be configured with:

!Image.jpg|align=center!
  • valid values for align attribute are 'left', 'right' and 'center'

Images can be configured with vertical and horizontal white space around them (and often should):

!Image.jpg|vspace=20,hspace=20!
  • image attributes are separated by comma

Tables

Use {ortable} for table definitions as follows:

{ortable}
 {tr}{th}Header 1{th} {th}Header 2{th}{tr}

 {tr}
   {td}Row 1, Column 1{td}
   {td}Row 1, Column 2{td}
 {tr}

 {tr}
   {td}Row 2, Column 1{td}
   {td}Row 2, Column 2{td}
 {tr}
{ortable}

Renders as :

Header 1 Header 2
Row 1, Column 1 Row 1, Column 2
Row 2, Column 1 Row 2, Column 2
  • {tr} stands for 'table row'
  • {th} stands for 'table header'
  • {td} stands for 'table data'

Non-Formatted Text

To enter text without any formatting or markup, use the following:

{noformat}Text is here{noformat}

Will render as:

Text is here, including {markup} and [links].

Formatting XML and Source Code

For XML, use the following:

{code:xml}
<absolute left = "80" top = "58" width = "640" height = "240">
  <web id = "6" src = "http://192.168.1.117/control/faststream.jpg?stream=full&fps=8.0" 
       username = "admin" password = "meinsm"/>
</absolute>
{code}

Will render as:

<absolute left = "80" top = "58" width = "640" height = "240">
  <web id = "6" src = "http://192.168.1.117/control/faststream.jpg?stream=full&fps=8.0" 
       username = "admin" password = "meinsm"/>
</absolute>

For Java source, use the following:

{code:java}
for (Element el : propertyElements)
    {

      // XML_ATTRIBUTENAME_VALUE defines the "value" attribute of "property" element
      // in the XML snippet, i.e. <property name = "foo" value = "bar"/>

      String propertyValue = el.getAttributeValue(CommandBuilder.XML_ATTRIBUTENAME_VALUE);

Escaping Links and Macros

Sometimes it is useful to include special characters [] or {} in your text. However, without escaping, the wiki will attempt to create a link of the former (which in all likelihood appears as red text since what is interpreted as link content will point to a non-existing page) or it will attempt to render the content of a macro in case of latter which will show up as an ugly error on the page.

To escape links or macros, use the backslash '\' character. So writing on the page markup:

\[parameter]

\{parameter}

Will render as expected:

[parameter]

{parameter}

Without wiki attempting to interpret the first as link or the second as macro.
